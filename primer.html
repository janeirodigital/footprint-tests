Semantic Web Applications interoperate by sharing semantics of terms and constellations of data structures. This primer introduces Footprints and describes how Semantic Web Applications use them to consistently interoperate over constellations of data structures.

This is a proposal for the structure and use of Footprints in Solid. It is possible that this or a derivative document will become a Solid specification.

Introduction

Expand explanation on exactly WHAT footprints are and WHY they are needed.

Realizing the value proposition of the Semantic Web requires manageable access to interoperable data. To enable interoperability, applications must be able to reliably locate, read and write like data to a Pod consistently. Solid access controls exploit the resource hierarchy to provide sensible intuition and management of nested resources.

Establishing intuitive access boundaries for data requires understanding:

  • the shapes that define and validate the content of the data
  • the structure of the resource hierarchy

Application interoperability depends on applications sharing semantics for relationships and data structures. RDF's foundation in unambiguous identifiers provides an infrastructure that allows for interoperability without specifically encouraging it. Within a single resource, shapes languages (e.g. ShEx and SHACL) provide machine-readable, enforceable data structure definitions. For applications that operate on constellations of resources, Footprints express the layout of those resources and associate them with their respective shapes.

A Footprint defines a prospective tree of related resources which can be read and written by applications. The footprint associates each of these resources with a shape. This allows one to treat a set of related resources as a whole, for access control, organization, validation and data migration purposes. When a new instance of that shape tree is needed for storage, the tree is stamped, creating an imprint.

Footprints are defined as an RDF graph structure and the behaviors of agents respecting that graph structure. These semantics can be implemented by a server, a client-side library, or a privilenged third-party helper application. Communication, including error reporting, is defined in terms of HTTP; allowing a single definition to define the behaviors for servers, clients and third-party applications.

Here, medicalNotes and astronomy are both imprints of the <#container> footprint. They are both LDP Containers and could contain any number of notes. In this example, xrays-2019-08, GP-2020-04-18 and blue-shift respectively are imprints of the <#note> footprint. Each contains a binary image and blue-shift has a citation.

The footprint is broken up into steps. The part of the footprint describing the citation has the subject node <#citation>. This step asserts that associated resources (e.g. cit-M33.ttl) must be LDP Resources, must have a name matching a URI template (i.e. must start with "cit-"), and must conform to the shape <CommonNote#Citation>.

The above resource hierarchy includes multiple imprints for medical and astronomy notes. Many operating systems make life simpler for users by providing a single place for specific data structures. This provides applications with the ability to store data without prompting the user for a storage target.

The example on the right implies a single location for all CommonNote notes. The name could come from the footprint name, from an application name, or be an opaque identifier invented by the system.

Should footprints include a step for their managed container? In the above footprint, the triples: 

<#container> foot:type ldp:Container ;
  foot:contents <#note> .

describe not an instance of a note but a container of such instances. Without these triples, the outer-most step (the one that is not the object of any foot:contents arcs) is <#note>. This makes it easier to, at a high-level, assert that the footprint is about Notes rather than containers of notes.

A footprint is a tree structure of steps which in turn describes a tree structure of Resources (effectively a hierarchy of directories and files). Each file or directory is described by step.

A step consists of:

  • an id: an addressable label for this step.
    As with all RDF structures, providing an id makes is possible to address this entity from outside the document.
  • a type: LDP type of the associated resource: BasicContainer OR Resource.
    Each each step in a footprint defines a resource in an LDP hierarchy. The type specifies whether the associated resources should be LDP:BasicContainers or LDP:Resources.
  • either of:
    • a label: label for a resource which will always be created;
      This is used when resources with specific names will appear in the resource hiearchy.
    • a uriTemplate: pattern for a resource to be matched against a POSTed Slug: header .
      This is used when the client gets to choose the name of a resource by including a Slug header in the request. could use a Link: header instead what is the problem we are solving here? not sure the stated problem is still relevant, but a new one likely exists in its place
  • any number of uses: identifies related footprints.
    These provide advice that the data in resources described by this step references another footprint. We need an example of this.
  • either of:
    • a shape: The RDF graph structure of any POSTed or PUT resource must conform to this shape;
      This allows footprints to ensure that the data in the resource hiearchy conforms to the expected schemas.
    • a mediaType: any non-RDF type must specify a media type or a media tree.
      This specifies the expected media type or media tree for non-RDF resources.
  • an optional contents: a list of nested steps.
    This captures the containership of a conforming resource hiearchy.

When POSTing to any Instance Container, there will be a corresponding step in the Footprint. If the step includes a shape, the POST must include a Link: rel="root" header to identify the node in the POSTed data that should conform to that shape.

When receiving a POST to a managed resource, an application POSTs to a Managed Container, the POD locates the corresponding step in the Footprint. If the step includes a shape, the POST body is parsed and the root node is tested for conformance. If it's invalid, the requested with a 422 Unprocessable Entity message. If it's valid or has no associated shape, the typical LDP processing will create an entity whose contents include the POSTed body.

If processing produces no errors, the POD will again Create any sub-Containers which have rdfs:labels in the Footprint. For instance, the GitHub Footprint step for {userName} has a nested Containers Subscriptionss (and many others).

Examples

This document describes the use of Footprints using three example tasks:

  • PhotoAlbum - collects photos or references to photos into sharable albums (footprint Turtle / JSON-LD, schema).
  • MultiCal - migrates calendar applications between two common formats, "Common" (footprint Turtle / JSON-LD, schema), and "Google" (footprint Turtle / JSON-LD, schema).
  • GitHub - a storage repository associating users or groups with repositories, issues and events (footprint Turtle / JSON-LD, schema).

The application directories described in this document are suffixed with "App" to make it easy to distinguish the App directory from the possibly similarly-named application and footprint.

Footprints can enable compatible applications to interoperate by identifying resources they can share. Conventions for indexing footprints and storing application metadata allow applications to find interoperable data and identify the applications that depend on or manipulate that data.
